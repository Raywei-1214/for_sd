# sd 流量优化方案

## 当前基线

- 以 `run_report_20260416_145058.json` 为例：
  - 60 个任务累计请求数：`55749`
  - 累计响应数：`52411`
  - 失败请求数：`2949`
  - 累计资源量：`1627.06 MB`
- 当前最耗流量的请求类型：
  - `script`
  - `image`
  - `xhr`
  - `fetch`
  - `stylesheet`

## 当前判断

- 流量大头不是 Notion、日志或截图。
- 流量主要烧在：
  - Dreamina 首页与注册页的脚本资源
  - 临时邮箱站点的整页加载与轮询刷新
  - 验证码阶段的邮箱正文预览与 Ajax 刷新

## 优化原则

- 先做可观测，再做省流。
- 优先改“重复整页加载”，谨慎拦截前端核心资源。
- 先做站点级轻刷新，再考虑全局资源拦截。

## 分阶段方案

### 第一阶段：低风险省流

- `open_home` 失败后减少重复重打首页。
- 优先用站点内轻刷新代替整页 `reload()`。
- 对验证码阶段做 provider 专用刷新：
  - `10minutemail.net` 优先调用站内 `updatemailbox()`
  - 仅在多轮仍无验证码时才整页刷新
- 报告继续保留网络统计，做前后对照。

### 第二阶段：中风险 A/B

- 只对临时邮箱页拦截 `image`。
- Dreamina 主站先不拦资源，只测邮箱站点。
- 对照指标：
  - 成功率
  - `wait_confirmation`
  - `fill_verification_code`
  - `network_transferred_megabytes`

### 第三阶段：高风险控制实验

- Dreamina 主站按域名做资源拦截白名单。
- 仅在前两阶段数据稳定后，再考虑：
  - `font`
  - 非关键第三方 `image`
  - 非关键 `media`
- 这一阶段必须做小样本 A/B，对照失败步骤分布后再决定是否保留。

## 当前优先项

1. 继续优化 `10minutemail.net` 的验证码链路。
2. 保留网络统计，按 20 / 60 / 100 样本做流量对照。
3. 先不要对 Dreamina 全站直接启用全局资源拦截。
